02
谁在做研究

三个 AI 角色对抗，一个人类掌控全局

日常的执行、审查、写作交给三个职责互斥的 Agent；人不做体力活，只做最关键的决策——定方向、批资源、控节奏。这套分工既保证产出，又保证可控。

PI
Principal Investigator

Director

方向决策、资源/GPU 管理、审批实验。不能自己跑实验——把自己当成"带一支 AI 团队的 PI"，按团队速度排程。

EX
Executor

Main Agent

调度 + 轻量规划。自己不做 SSH / 写代码 / 读日志——重活派给一次性 Worker，实验 registry 为唯一事实来源。

RV
Independent Auditor

Reviewer

只读权限，与执行并行穿插。独立复核每个结论，返回 info / warning / critical，critical 直接暂停项目。

Human-in-control · 人掌控全局

人是"PI 的 PI"，不是旁观者

系统不是失控黑箱：立项与方向由人提出与拍板；动用 实验、GPU、密钥等关键资源必须人工审批；预算与节奏由人掌控，可随时介入、暂停或要求 pivot——全程在看板/群里可见、可干预。

为什么"对抗式"

产出与验证，物理隔离

执行体和审查体是两个不同进程、不同工具权限、不同 prompt。执行 Agent 无法伪造审查 Reviewer 看到的东西——Reviewer 独立读取原始 nvidia-smi、训练日志与 W&B 指标。这是系统可信度的根基。

04
Harness 护城河 · 前瞻能力（1 / 3）

Dream：让系统拥有"预见结果"的前瞻能力

普通 Agent 只能"跑完才知道好坏"。Dream 让系统在开跑之前，就异步推演一条研究路线会走到哪——预见结局、预见审稿意见，从而带着前瞻去规划，而不是被动地等结果出来再反应。

怎么做到的

两阶段"想象" → 写成论文 → 真盲审

① Tier 定义一次推理切出 best / median / worst 三档结局
② 逐档 elaboration顺序细化成"想象中的实验设置与数字"
③ 写成完整论文Paper Agent 据此产出整篇 LaTeX
④ 送进真实盲审哈希软链接藏掉档位名，审稿人不知在读"最好情况"

带 simple / full 两档 review、跨次趋势追踪、SHA-256 智能跳过（无新信息不重算）；数据落 per-project YAML。

前瞻性带来什么

从"事后反应"升级到"事前规划"

  • 预见结局：开跑前就看清这条路的 best / median / worst 大概落在什么水位
  • 预见审稿意见：提前知道审稿人会卡在哪（novelty / 严谨性…），据此调整 framing 与路线
  • 主动规划而非被动等结果：Director 拿着这份"预测"排兵布阵，决定主攻哪条线、哪条线先放一放
  • 已被实测验证：Dream 的预测与最终真实审稿高度吻合——simgate 项目 Dream 早就点出 novelty 是瓶颈，最终 EMNLP 两位审稿人独立给出的瓶颈完全一致
一句话：别人的 Agent 只能"跑完才知道结果"；我们的系统能预见结果——开跑之前就拿到一份"预测 + 模拟审稿"，让研究有了前瞻和规划，而不是盲目试错。（算力省下来只是顺带的红利。）

05
Harness 护城河 · 领域知识（2 / 3）

蒸馏体系：给系统注入一个领域的"内行知识"

AI 不天然知道某个子领域的人怎么做 baseline、怎么画图、怎么讲故事。蒸馏体系把一个领域的文献与过往经验，提炼成系统可复用、可累积的领域知识。

第一层 · 项目级文献蒸馏

逐篇深读 + 跨篇归纳

  • 沿 Semantic Scholar 引用图谱采集子领域文献（文献图谱 + 关键词 + 推荐 + 前后向引用）
  • 对核心论文做 全文 + 配图深读（最多 8 篇，结构化成 YAML）
  • 跨篇归纳出两套范式：研究方法范式（baseline/指标/数据/消融）+ 写作范式（叙事结构/Figure-1/主结果表）
  • 落成项目专属、带版本的 skill；每条模式带 支持度（N/M 篇）+ 来源 + 置信度——是归纳，不是摘要
第二层 · 知识代谢引擎（Distill Hub）

跨项目累积，越用越厚

  • 把每个项目跑出的经验沉淀成四级抽象：事件 → 观察 → 模式 → 技能
  • 按 project → domain → global 逐级提升（一个模式在 ≥3 个项目复现→升 domain；≥2 个 domain→升 global）
  • 新项目启动即继承前人沉淀，跨方法论 / 写作 / 工程 / 文献四个维度复用
  • 双环飞轮：内环 执行→事件→反思→技能→执行；外环 轨迹分析→训练数据→微调
为什么是护城河：领域知识是越跑越厚的累积资产——跑的项目越多，沉淀的"内行经验"越多，后来的项目起点越高。规模本身构成壁垒。

06
Harness 护城河 · 长程自主（3 / 3）

长任务循环：让 AI 真能连续工作数天而不掉线

一个研究项目要跑数天。让一群 Agent 在数天里不卡死、不因上下文写满而"失忆"，是工程上最难的部分。两个机制兜住它。

机制一 · 催促 / Keepalive

自主 PI 看门狗：不让 GPU 空烧

Director 跑在后台、输出对用户不可见；一旦它发呆，几十张 GPU 就在空转烧钱。守护进程检测到静默后：

  • 自动收集实时健康信号：失败 Worker / 待审批 / 卡住 cron / GPU 空闲
  • 按 鼓励 → 提醒 → 审计 → 告警 分级，生成一条带上下文的催促
  • 逼 Director 用 ACK / WAIT / BLOCKED 明确回应
  • 连续两次不理会，自动升级到运维群叫人
机制二 · Ralph Loop

上下文轮转：近乎无界的自主时长

单个上下文窗口装不下数天的工作，且会随写满逐渐"退化变笨"。Ralph Loop 做上下文轮转：

  • 触顶（硬限 1000 消息 / 32h）时，Director 批准重置
  • 旧会话写一封 "给下一个自己的交接信"
  • 新会话带 交接信 + 从 memory 抽取的关键约束 + "先读实时 registry" 冷启动
  • 全程 append-only 时间线落盘，重启不丢 loop 计数
一句话：这两个机制是"7×24 运转的 AI 研究团队"能成立的底层保证——没有它们，长项目会因卡死或失忆而中途崩掉。

核心 · 旗舰

Apex Research 模式

自动产出论文。从 Idea 到投稿全自动闭环，Paper Agent 跑完整 5 步流水线。面向"会议投稿"的旗舰形态。

Research 模式

方向性探索 / 实验，不必跑完整论文。产物多样化——技术报告、消融结论、可行性判断。团队内部用它跑实验已经非常实用。

  • 产物：可独立核验的榜单成绩
  • 已完整跑通 3 个 Benchmark

项目概览：AI4Sci-Bench 是什么

AI4Sci-Bench 是面向 AI for Science 领域的项目级科学计算任务评测基准。区别于 HumanEval、SWE-bench 等聚焦软件工程的 benchmark，AI4Sci-Bench 专注于评估 AI Agent 在真实科研场景中的算法实现、数值计算与科学推理能力。覆盖从天文学到机器人学的 13 个学科领域，包含 97 个专家级任务，支持 7 种主流 Agent Harness 的即插即用接入。

* 架构图自上而下：模型层可接入任意 LLM → Harness 层支持 7 种主流 Agent 框架（厂商原生 + 开源）→ 统一评分引擎 → 13 个学科领域 97 个专家级任务。所有层级完全解耦，新增 Harness 或新增 Task 无需修改其他层代码。

03

四级 Prompt 难度：B1 → B4 渐进式信息削减

每个任务配备 4 个难度等级的 prompt，从完整算法说明到裸需求描述再到含干扰项的测试。不只是"题目变短"，而是系统性地考察 Agent 的不同认知能力。

B1 · DETAILED
100%
信息量（基线）
完整的算法说明书
包含显式数学公式、伪代码、离散化方案、求解器选项、参数范围、验证指标。
例：物理任务 274 行
含 13 步积分流程 + 核函数导数 + Poisson 离散格式 + 边界条件
B2 · INTERMEDIATE
~60%
信息量
高层流程概述
保留步骤名称和方法选择，但移除所有显式公式、参数细节和实现提示。
例：同一物理任务 59 行
12 步流程名 + 无公式 + 无核函数定义
B3 · MINIMAL
~40%
信息量
裸需求 + 输出契约
只给问题描述和输出文件格式，不暴露任何方法提示。Agent 必须自主选择算法。
例：同一物理任务 33 行
仅概念名 + 输出规格 + 无任何流程指引
B4 · DISTRACTOR
B3 + noise
信息量
B3 全文 + 干扰段落
附加事实正确但与解题无关的背景知识，模拟真实论文中的信息噪声。
例：同一物理任务 53 行
B3 + 历史综述(1931) + 竞争方法(FLIP/MPM)

为什么需要 4 个 Level

1. 测试不同认知能力

B1 测实现能力（照着食谱做菜），B3 测科学推理（只知道要做什么菜、自己想做法）。两者是完全不同的能力维度。一个模型可能 B1 高分但 B3 崩溃——说明它擅长执行但不会推理。

2. 量化 Prompt 敏感性

B1→B3 分差揭示模型对指令依赖程度。Opus 4.8 衰减仅 -0.3，Gemini 3.1 Pro 衰减 -13.5。顶级模型的标志是对 prompt 详细程度不敏感。

3. 抗干扰能力检验

B4 模拟科研人员的真实工作环境——论文里充斥着相关但无用的信息。模型需要从噪声中提取关键约束，而非被旁枝末节带偏。

🔵 B1 · 完整算法说明（77 行） 给公式、给参数、给求解流程 ▶ 
── 英文原文（关键段落摘录）──

The hidden physical problem is the steady 2D
incompressible lid-driven cavity on a square domain.
- bc_primary.npy is the full-grid horizontal
  boundary-velocity array.
- The moving lid is encoded by nonzero horizontal
  velocity values on the top wall.

Numerical guidance:
- Solve laplacian(psi) = -omega with homogeneous
  Dirichlet boundary conditions on psi.
- Recover velocity: u = dpsi/dy, v = -dpsi/dx.
- Use the Ghia-style wall-vorticity closure:
  omega_wall = -(4*psi_adj - 0.5*psi_next)/h^2
  omega_top  = ... - 3*U_lid/h
- Use second-order upwind convection discretization,
  with RK2-style update.
- Conservative time step (CFL + diffusion).
- Use coarse-to-fine initialization and Reynolds
  continuation before the final solve.

隐含物理问题是正方形区域上的稳态二维

- 移动盖板由顶壁的非零水平速度编码。

数值指引：

结果分析：闭源领跑，国产追近，Harness 决定下限

从 15 种"模型 × Harness"组合的均分中可以读出三条清晰的信号：闭源旗舰仍处于第一梯队，但优势并不绝对；国产模型在原生 Harness 加持下已能贴近第二梯队上沿；Harness/Adapter 的选择对最终分数的影响，往往不亚于换模型本身。

第一梯队 · 闭源旗舰
48.8 / 45.6 / 35.7

Claude Fable 5 与 Opus 4.8 在 CC-CLI 上分别拿到 48.8 / 45.6，Gemini 3.1 Pro 在 OpenHands 上达到 35.7，三个闭源旗舰占据榜单前三。Claude 系列在 9 个非管控任务上接近天花板，但 4 个 biology/chemistry 任务因安全管控直接归零——能力没问题，是 policy 问题。

第二梯队 · 国产追近
33.1 / 32.1 / 31.3

Kimi K2.7 + Kimi CLI（33.1）已超过 Gemini 与 GLM-5.2 之外的所有 OpenHands 配置，仅与 Gemini 3.1 Pro 差 2.6 分。GLM-5.2 + ZCode（32.1）、Kimi K2.6 + Kimi CLI（31.3）紧随其后。在 原生 CLI Harness 加持下，国产模型与闭源第二梯队的差距已缩到 3 分以内，部分维度反超（如 K2.7 B2 比 GLM-5.2 B2 高 12.1 分）。

关键发现 · Harness 即放大器
+10.7 分

同一个 Kimi K2.6，从 OpenHands 换到原生 Kimi CLI，均分从 22.4 → 31.3，纯换 Harness 涨 8.9 分；K2.7 从 OpenHands 27.7 → Kimi CLI 33.1，再升 5.4 分。15 种组合中头尾相差 30.8 分，其中"换 Harness"贡献的差值时常超过"换模型"。这是 ASI4Sci-Bench 与其他 benchmark 不一样的地方——它评的是「模型 × Agent 框架」整体，而不是孤立的模型能力。

一句话总结：当下 AI for Science 评测呈现"闭源旗舰领先、国产模型贴身追赶、Harness 主导下限"的三层格局。国产代表（Kimi K2.7、GLM-5.2、MiMo V2.5 Pro）在搭配自家原生 CLI 时表现并不逊色，差距正在被工程层面的优化（原生工具调用、上下文管理、思考链格式）持续抹平。

为什么是现在

AI for Science 正在从论文验证走向真实科研工具。各大厂商的 Coding Agent 能力突飞猛进，但没有人在系统性地评估它们在科学计算场景下的真实能力。现有 benchmark（HumanEval、SWE-bench 等）聚焦软件工程，无法覆盖科学计算的核心挑战：数值精度、物理约束、算法收敛性。AI4Sci-Bench 填补了这个空白。

08

自进化闭环：评测 → 轨迹 → 训练 → 再评测

AI4Sci-Bench 不只是一个 benchmark，而是一套自成体系的科学智能体进化环境。模型在任务中产生轨迹，优质轨迹训练出更强模型，更强模型解锁更多任务、产出更好轨迹——每一轮迭代都让系统整体变强。

四步闭环：每一轮都让系统变强

1
评测（Evaluate）
将当前模型接入 AI4Sci-Bench，在 97+ 个科学任务 × 4 个 prompt 等级上全量评测。评分引擎自动产出每任务、每 B-level 的精细得分。
2
轨迹收集（Collect）
所有模型运行过程中的完整轨迹（思考链、工具调用、代码生成、调试过程）被自动录制。筛选高分 run 的轨迹作为优质训练数据，低分 run 的轨迹用于错误分析。
3
模型训练（Train）
用优质轨迹对模型做 SFT / RLHF 微调，针对性增强科学推理、数值计算、算法实现能力。训练信号直接来自真实科研任务，而非合成数据。
4
再评测（Re-evaluate）
训练后的新模型重新跑全量评测。与上一轮对比，精确定位哪些学科、哪些任务类型获得了提升，哪些仍是短板——然后进入下一轮循环。
为什么这是一个「自进化」系统：传统 benchmark 是静态的——出题、测试、结束。AI4Sci-Bench 是动态的——评测本身产出训练数据，训练产出更强的模型，更强的模型解锁更难的任务、产出更高质量的轨迹。系统的每一个组件都在让其他组件变强。

02
我们要解决的问题

AI 已经会写代码，但"一个人能管好几个 AI"还没人帮我们解决

编码智能体这一年能力突飞猛进，但团队里大多数人还停留在"开一个窗口、问一句、等一句"的用法。真正的瓶颈，已经从"模型够不够强"，变成了"一个人到底能同时驱动多少个智能体"。CCM 就是为补上这块短板而生的内部工具。

现状的四个痛点

串行、值守、排队、不沉淀

  • 串行使用，能力闲置：一次只盯一个对话，智能体大量时间在等人，强大的模型能力被白白浪费。
  • 人工搬运，全程值守：建分支、跑测试、合并、解决冲突、写记录——每步都要人手动衔接，注意力被切碎。
  • 无法并行，项目排队：多个项目只能一个一个推进，个人产能被锁死在"单线程"。
  • 经验不沉淀，重复踩坑：每次从零开始，上次的教训这次又踩一遍，团队知识无法累积。

平台纵深与演进

从单机，到舰队，再到团队

CCM 的能力边界还在快速扩张。三件正在发生的事，决定了它能走多远——一个已成规模、一个初见雏形、一个蓄势待发。

W
已具备 · 规模化底座

弹性 Worker 系统

CCM 从不局限于一台机器。它能按需在云端自动拉起一批 Worker 工作机，组成可弹性伸缩的执行集群——任务在 Manager 与 Worker 之间自动分发、迁移、回收，单机算力上限不再是天花板。

本次数据中的 62 个工作实例、25 路同时并行，正是这套弹性系统撑起来的。

初步落地 · 向上生长

CCM 已具备搭建自定义 Harness（智能体运行框架）的能力，目前已有初步实现。这让平台不被单一形态绑定，未来可承载更多类型的智能体与运行模式。
